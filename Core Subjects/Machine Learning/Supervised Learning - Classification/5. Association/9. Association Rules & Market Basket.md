## ASSOCIATION RULES & MARKET BASKET ANALYSIS

**Definition:** An unsupervised learning technique that discovers interesting relationships, patterns, and associations between items in large transactional databases. It identifies rules of the form "If A, then B" (A → B), meaning customers who bought item A are likely to buy item B. These rules reveal hidden patterns in data that can be used for cross-selling, product placement, and recommendation systems.

**Used For:**

- Market basket analysis (retail)
- Product recommendation systems
- Cross-selling and up-selling strategies
- Store layout optimization
- Promotional bundling
- Web usage mining (clickstream analysis)
- Medical diagnosis (symptom associations)
- Fraud detection patterns
- Telecommunications (service bundling)

**Cases of Application:**

- Retail: "Customers who buy diapers also buy beer"
- E-commerce: "People who viewed X also bought Y"
- Netflix/Streaming: "Users who watched A also watched B"
- Healthcare: "Patients with symptoms X, Y often have disease Z"
- Insurance: Fraud pattern detection
- Banking: Service bundling, credit card usage patterns
- Telecommunications: Call pattern analysis
- Restaurant: Menu item combinations
- Web analytics: Page visit sequences
- Supply chain: Product demand patterns

**Prerequisites:**

- Transactional data (sets of items purchased together)
- Binary data format (item present/absent in transaction)
- Sufficient number of transactions (typically 100+)
- Items should have reasonable frequency (not too rare)
- No need for labeled data (unsupervised)
- Data should be in "basket" format (one row per transaction)
- Handle missing values (treated as item not purchased)

**Performance Measures:**

- **Support:** Frequency of itemset in dataset
- **Confidence:** Likelihood of B given A
- **Lift:** Strength of association (> 1 is good)
- **Conviction:** Dependency measure
- **Leverage:** Difference from independence
- **Number of rules generated**
- **Average rule length**
- **Execution time**
- **Business value/actionability of rules**

## CORE CONCEPTS

### **1. Transaction Database**

**Definition:** A collection of transactions, where each transaction contains a set of items.

**Example:**

```
Transaction Database:

TID | Items Purchased
----|------------------
T1  | {Bread, Milk}
T2  | {Bread, Diaper, Beer, Eggs}
T3  | {Milk, Diaper, Beer, Coke}
T4  | {Bread, Milk, Diaper, Beer}
T5  | {Bread, Milk, Diaper, Coke}
```

**Format Requirements:**

- Each row = one transaction
- Each transaction = set of items
- Binary: item either present (1) or absent (0)

---

### **2. Itemset**

**Definition:** A collection of one or more items.

**Types:**

**k-itemset:** Itemset containing k items

**Examples:**

```
1-itemset: {Bread}, {Milk}, {Beer}
2-itemset: {Bread, Milk}, {Beer, Diaper}
3-itemset: {Bread, Milk, Diaper}
```

**Frequent Itemset:** Itemset whose support ≥ minimum support threshold

---

### **3. Association Rule**

**Definition:** Implication of form A → B where A and B are itemsets with no common items (A ∩ B = ∅).

**Components:**

- **Antecedent (LHS):** Items on left side (A)
- **Consequent (RHS):** Items on right side (B)

**Example:**

```
Rule: {Bread, Butter} → {Milk}

Interpretation: 
"Customers who buy Bread AND Butter 
are likely to buy Milk"

Antecedent: {Bread, Butter}
Consequent: {Milk}
```

---

## KEY METRICS

### **1. SUPPORT**

**Definition:** Proportion of transactions containing the itemset.

**Formula:**

```
Support(A) = (Transactions containing A) / (Total transactions)

Support(A → B) = Support(A ∪ B)
               = (Transactions containing both A and B) / (Total transactions)
```

**Range:** [0, 1] or [0%, 100%]

**Example:**

```
Database: 5 transactions
Transactions with {Bread, Milk}: 3

Support({Bread, Milk}) = 3/5 = 0.6 = 60%
```

**Interpretation:**

- High support: Itemset occurs frequently (popular items)
- Low support: Itemset is rare
- Minimum support threshold: Filter out infrequent itemsets

**Typical Thresholds:**

- Retail: 0.1% - 1% (low threshold, many items)
- Small dataset: 10% - 20%
- Large dataset: 0.01% - 0.1%

**Why Important:**

- Filters rare/noise patterns
- Reduces computational complexity
- Focuses on significant patterns

---

### **2. CONFIDENCE**

**Definition:** Likelihood of B occurring given A has occurred. Measures reliability of the rule.

**Formula:**

```
Confidence(A → B) = Support(A ∪ B) / Support(A)
                  = P(B | A)

Probability of B given A
```

**Range:** [0, 1] or [0%, 100%]

**Example:**

```
Transactions with {Bread}: 4
Transactions with {Bread, Milk}: 3

Confidence({Bread} → {Milk}) = 3/4 = 0.75 = 75%

"75% of customers who buy Bread also buy Milk"
```

**Interpretation:**

- 1.0 (100%): Perfect rule - B always occurs with A
- 0.75 (75%): Strong association - B often occurs with A
- 0.5 (50%): Weak association
- Close to 0: Rule not useful

**Typical Thresholds:**

- Minimum confidence: 50% - 80%
- Depends on business needs

**Limitation:**

- Can be misleading if B is very popular (high baseline)
- Example: If 90% of all customers buy milk anyway, 75% confidence is not impressive

---

### **3. LIFT**

**Definition:** Ratio of observed support to expected support if A and B were independent. Measures how much more likely B is purchased when A is purchased, compared to B being purchased in general.

**Formula:**

```
Lift(A → B) = Confidence(A → B) / Support(B)
            = Support(A ∪ B) / (Support(A) × Support(B))
            = P(A ∩ B) / (P(A) × P(B))
```

**Range:** [0, ∞)

**Interpretation:**

**Lift = 1:** A and B are independent (no association)

- Buying A doesn't affect likelihood of buying B
- Rule is not useful

**Lift > 1:** Positive correlation (A and B appear together more than expected)

- Buying A increases likelihood of buying B
- **Good association rule**
- Lift = 2: B is twice as likely with A
- Lift = 3: B is three times as likely with A

**Lift < 1:** Negative correlation (A and B are substitutes)

- Buying A decreases likelihood of buying B
- Items may be substitutes/competitors

**Example:**

```
Support({Bread}) = 0.4 (40%)
Support({Milk}) = 0.5 (50%)
Support({Bread, Milk}) = 0.3 (30%)

Confidence({Bread} → {Milk}) = 0.3/0.4 = 0.75 = 75%

Lift({Bread} → {Milk}) = 0.75 / 0.5 = 1.5

Interpretation:
"Customers who buy Bread are 1.5 times (50%) more likely 
to buy Milk compared to customers in general"

Expected support if independent: 0.4 × 0.5 = 0.2 (20%)
Actual support: 0.3 (30%)
Lift: 30% / 20% = 1.5
```

**Typical Thresholds:**

- Minimum lift: 1.0 - 1.5
- Strong association: Lift > 2
- Very strong: Lift > 3

**Why Lift > Confidence:**

- Lift accounts for item popularity
- Confidence can be high just because consequent is popular
- Lift shows **true correlation**

---

### **4. CONVICTION**

**Definition:** Measures the dependency of B on A. How much more often B would be expected incorrectly if the rule didn't hold.

**Formula:**

```
Conviction(A → B) = (1 - Support(B)) / (1 - Confidence(A → B))
                  = P(A) × P(¬B) / P(A ∩ ¬B)
```

**Range:** [0, ∞)

**Interpretation:**

**Conviction = 1:** A and B are independent **Conviction > 1:** A and B dependent (good rule)

- Higher values = stronger association **Conviction = ∞:** Perfect implication (confidence = 1)

**Example:**

```
Support({Milk}) = 0.5
Confidence({Bread} → {Milk}) = 0.75

Conviction = (1 - 0.5) / (1 - 0.75)
           = 0.5 / 0.25
           = 2.0

"The rule {Bread} → {Milk} would be incorrect 
twice as often without the association"
```

**Use When:**

- Want to measure implication strength
- Complement to lift
- Asymmetric measure (A → B ≠ B → A)

---

### **5. LEVERAGE**

**Definition:** Difference between observed frequency and expected frequency if independent.

**Formula:**

```
Leverage(A → B) = Support(A ∪ B) - (Support(A) × Support(B))
                = P(A ∩ B) - P(A) × P(B)
```

**Range:** [-1, 1]

**Interpretation:**

- **Leverage = 0:** Independent
- **Leverage > 0:** Positive association
- **Leverage < 0:** Negative association
- Higher absolute value = stronger association

**Example:**

```
Support({Bread, Milk}) = 0.3
Support({Bread}) = 0.4
Support({Milk}) = 0.5

Leverage = 0.3 - (0.4 × 0.5)
         = 0.3 - 0.2
         = 0.1

Positive leverage: items appear together more than expected
```

---

### **6. KULCZYNSKI MEASURE**

**Definition:** Average of two conditional probabilities.

**Formula:**

```
Kulc(A → B) = 0.5 × (P(B|A) + P(A|B))
            = 0.5 × (Conf(A→B) + Conf(B→A))
```

**Use:** Symmetric measure of association

---

### **7. COSINE SIMILARITY**

**Definition:** Geometric measure of association.

**Formula:**

```
Cosine(A, B) = Support(A ∪ B) / √(Support(A) × Support(B))
```

**Range:** [0, 1]

---

### **METRIC COMPARISON:**

|Metric|Type|Range|Symmetric?|Accounts for Item Popularity?|Best For|
|---|---|---|---|---|---|
|**Support**|Frequency|[0, 1]|Yes|No|Filtering frequent items|
|**Confidence**|Conditional|[0, 1]|No|No|Rule reliability|
|**Lift**|Correlation|[0, ∞)|Yes|**Yes**|**True association strength**|
|**Conviction**|Dependency|[0, ∞)|No|Yes|Implication strength|
|**Leverage**|Difference|[-1, 1]|Yes|Yes|Deviation from independence|

**Most Important:** Support, Confidence, Lift

## APRIORI ALGORITHM

**Definition:** Classic algorithm for mining frequent itemsets and generating association rules. Uses level-wise search and "Apriori property" to prune search space.

### **Apriori Property (Anti-monotonicity):**

**Key Principle:** "If an itemset is frequent, then all of its subsets must also be frequent"

**Contrapositive:** "If an itemset is infrequent, then all of its supersets must be infrequent"

**Example:**

```
If {Bread, Milk, Butter} is infrequent,
then {Bread, Milk, Butter, Eggs} must also be infrequent

No need to check larger itemsets!
```

**Why Important:**

- Prunes search space dramatically
- Reduces computational complexity
- Makes algorithm tractable

---

### **APRIORI ALGORITHM STEPS:**

**INPUT:**

- Transaction database D
- Minimum support threshold (min_sup)
- Minimum confidence threshold (min_conf)

**OUTPUT:**

- Set of association rules meeting thresholds

**STEPS:**

#### **Phase 1: Find Frequent Itemsets**

**Step 1: Generate Candidate 1-itemsets (C₁)**

```
Scan database
Count frequency of each item
```

**Step 2: Find Frequent 1-itemsets (L₁)**

```
L₁ = {items with support ≥ min_sup}
Prune infrequent items
```

**Step 3: Generate Candidate 2-itemsets (C₂)**

```
Join L₁ with L₁ to create pairs
C₂ = {combinations of items from L₁}
```

**Step 4: Find Frequent 2-itemsets (L₂)**

```
Scan database
Count support for each candidate in C₂
L₂ = {itemsets in C₂ with support ≥ min_sup}
```

**Step 5: Repeat for k-itemsets**

```
For k = 3, 4, 5, ...
   Cₖ = generate_candidates(Lₖ₋₁)  [Join step]
   Cₖ = prune(Cₖ)                   [Prune step]
   Lₖ = {candidates with support ≥ min_sup}
   
   Stop when Lₖ is empty
```

**Join Step:**

```
Join Lₖ₋₁ with Lₖ₋₁ where first k-2 items are same

Example (k=3):
L₂ = {{A,B}, {A,C}, {A,D}, {B,C}, {B,D}}

Join {A,B} and {A,C} → {A,B,C}  ✓ (first item same)
Join {A,B} and {B,C} → ✗ (first items different)
Join {A,C} and {A,D} → {A,C,D}  ✓

C₃ = {{A,B,C}, {A,C,D}, {A,B,D}, {B,C,D}}
```

**Prune Step:**

```
Remove candidates with infrequent subsets

If {A,B,C} is candidate:
Check if {A,B}, {A,C}, {B,C} are all in Lₖ₋₁
If any subset missing, remove {A,B,C} from Cₖ
```

#### **Phase 2: Generate Association Rules**

**For each frequent itemset F:**

**Step 1:** Generate all non-empty subsets of F

**Step 2:** For each subset S:

```
Rule: S → (F - S)
Calculate confidence = Support(F) / Support(S)

If confidence ≥ min_conf:
   Output rule with metrics
```

**Example:**

```
Frequent itemset: {A, B, C}
Support({A,B,C}) = 0.3

Possible rules:
{A} → {B,C}        Conf = 0.3 / Support({A})
{B} → {A,C}        Conf = 0.3 / Support({B})
{C} → {A,B}        Conf = 0.3 / Support({C})
{A,B} → {C}        Conf = 0.3 / Support({A,B})
{A,C} → {B}        Conf = 0.3 / Support({A,C})
{B,C} → {A}        Conf = 0.3 / Support({B,C})

Keep rules where confidence ≥ min_conf
```

---

### **APRIORI EXAMPLE (STEP-BY-STEP):**

**Transaction Database:**

```
TID | Items
----|------------------
T1  | {A, B, C}
T2  | {A, C}
T3  | {B, C}
T4  | {A, B, C}
T5  | {A, B}

Total transactions: 5
Minimum support: 40% (2/5)
Minimum confidence: 60%
```

**Phase 1: Find Frequent Itemsets**

**Iteration 1:**

```
C₁ (1-itemsets):
Item | Count | Support
-----|-------|--------
A    | 4     | 80%  ✓
B    | 4     | 80%  ✓
C    | 4     | 80%  ✓

L₁ = {A, B, C}  (all meet min_sup)
```

**Iteration 2:**

```
C₂ (candidate 2-itemsets from L₁):
{A,B}, {A,C}, {B,C}

Count support:
Itemset | Transactions | Support
--------|--------------|--------
{A,B}   | T1,T4,T5     | 60%  ✓
{A,C}   | T1,T2,T4     | 60%  ✓
{B,C}   | T1,T3,T4     | 60%  ✓

L₂ = {{A,B}, {A,C}, {B,C}}
```

**Iteration 3:**

```
C₃ (join L₂ with L₂):
Join {A,B} and {A,C} → {A,B,C}
Join {A,B} and {B,C} → not valid (different first items)
Join {A,C} and {B,C} → not valid

C₃ = {{A,B,C}}

Prune step:
Check subsets of {A,B,C}: {A,B}, {A,C}, {B,C}
All are in L₂ ✓, so keep {A,B,C}

Count support:
{A,B,C} in T1, T4 → Support = 40% ✓

L₃ = {{A,B,C}}
```

**Iteration 4:**

```
Try to generate C₄ from L₃
Only one itemset in L₃, cannot join
Stop.

Frequent itemsets found:
L₁ = {{A}, {B}, {C}}
L₂ = {{A,B}, {A,C}, {B,C}}
L₃ = {{A,B,C}}
```

**Phase 2: Generate Rules**

**From {A,B,C} (support = 40%):**

```
Rule: {A} → {B,C}
Confidence = Support({A,B,C}) / Support({A})
           = 0.4 / 0.8 = 0.5 = 50%
50% < 60% ✗ Reject

Rule: {B} → {A,C}
Confidence = 0.4 / 0.8 = 50% ✗ Reject

Rule: {C} → {A,B}
Confidence = 0.4 / 0.8 = 50% ✗ Reject

Rule: {A,B} → {C}
Confidence = 0.4 / 0.6 = 0.667 = 66.7% ✓
Lift = 0.667 / 0.8 = 0.83 < 1 ✗ (negative correlation)

Rule: {A,C} → {B}
Confidence = 0.4 / 0.6 = 66.7% ✓
Lift = 0.667 / 0.8 = 0.83 ✗

Rule: {B,C} → {A}
Confidence = 0.4 / 0.6 = 66.7% ✓
Lift = 0.667 / 0.8 = 0.83 ✗
```

**From {A,B} (support = 60%):**

```
Rule: {A} → {B}
Confidence = 0.6 / 0.8 = 75% ✓
Lift = 0.75 / 0.8 = 0.94 ✗

Rule: {B} → {A}
Confidence = 0.6 / 0.8 = 75% ✓
Lift = 0.75 / 0.8 = 0.94 ✗
```

**Similar for other 2-itemsets...**

**Final Rules (meeting both confidence and lift > 1 criteria):** In this small example, no rules meet all criteria. With different data or lower thresholds, we would find useful rules.

### **APRIORI COMPLEXITY:**

**Time Complexity:**

```
O(2^n) in worst case (n = number of items)

In practice: Much faster due to pruning
- Database scans: O(k × |D|) where k = max itemset size
- Candidate generation: O(|L|²)
```

**Space Complexity:**

```
O(|C|) where |C| = number of candidate itemsets

Dominated by storing candidates and frequent itemsets
```

**Optimizations:**

- Hash-based pruning
- Transaction reduction
- Partitioning
- Sampling

## FP-GROWTH ALGORITHM

**Definition:** Frequent Pattern Growth algorithm. More efficient alternative to Apriori that doesn't generate candidate itemsets.

### **Key Advantages over Apriori:**

1. **No candidate generation:** Builds compressed data structure
2. **Only 2 database scans:** vs multiple scans in Apriori
3. **Faster:** Especially for large databases
4. **Less memory:** Compressed FP-tree structure

### **Algorithm Overview:**

**Step 1: Build FP-Tree**

- Scan database once to find frequent 1-itemsets
- Sort items by frequency (descending)
- Scan database again, insert transactions into tree

**Step 2: Mine FP-Tree**

- Start with least frequent item
- Find conditional pattern base
- Build conditional FP-tree
- Recursively mine conditional trees

### **When to Use FP-Growth vs Apriori:**

**Use FP-Growth when:**

- Large database (millions of transactions)
- Need faster execution
- Have memory for FP-tree
- Long transactions

**Use Apriori when:**

- Small database (simple to implement)
- Teaching/understanding concepts
- Need rule generation flexibility
- Memory constrained (simpler structure)

---

## ECLAT ALGORITHM

**Definition:** Equivalence Class Clustering and bottom-up Lattice Traversal. Uses depth-first search and vertical data format.

**Key Difference:**

- Apriori: Horizontal data format (TID → items)
- ECLAT: Vertical data format (Item → TIDs)

**Vertical Format:**

```
Item | Transaction IDs
-----|------------------
A    | {T1, T2, T4, T5}
B    | {T1, T3, T4, T5}
C    | {T1, T2, T3, T4}
```

**Advantages:**

- Fast intersection of TID sets
- No candidate generation
- Suitable for sparse datasets

**Disadvantages:**

- Memory intensive for dense datasets
- Diffset operations complex

**Basic Workflow:**

```python
import pandas as pd
from mlxtend.frequent_patterns import apriori, association_rules
from mlxtend.preprocessing import TransactionEncoder

# Sample transaction data
transactions = [
    ['Milk', 'Bread', 'Butter'],
    ['Milk', 'Bread', 'Eggs'],
    ['Bread', 'Butter'],
    ['Milk', 'Bread', 'Butter', 'Eggs'],
    ['Milk', 'Eggs']
]

# Convert to one-hot encoded format
te = TransactionEncoder()
te_ary = te.fit(transactions).transform(transactions)
df = pd.DataFrame(te_ary, columns=te.columns_)

print(df)
#    Bread  Butter   Eggs   Milk
# 0   True    True  False   True
# 1   True   False   True   True
# 2   True    True  False  False
# 3   True    True   True   True
# 4  False   False   True   True

# Find frequent itemsets using Apriori
frequent_itemsets = apriori(df, min_support=0.4, use_colnames=True)
print(frequent_itemsets)

# Generate association rules
rules = association_rules(frequent_itemsets, metric="confidence", min_threshold=0.6)

# Add lift
rules = association_rules(frequent_itemsets, metric="lift", min_threshold=1.0)

print(rules[['antecedents', 'consequents', 'support', 'confidence', 'lift']])
```

import pandas as pd
from mlxtend.frequent_patterns import apriori, association_rules

# Create sample dataset
data = {
    'Milk': [1, 1, 0, 1, 1, 0, 1, 1],
    'Bread': [1, 1, 1, 1, 1, 0, 0, 1],

## CHOOSING THRESHOLDS

### **Minimum Support:**

**Too High:**

- Miss interesting rare patterns
- Only obvious, trivial rules

**Too Low:**

- Too many rules (overwhelming)
- Computationally expensive
- Many spurious/noise patterns

**Guidelines:**

**Large Retail Dataset (millions of transactions):**

```
min_support = 0.001 - 0.01 (0.1% - 1%)
Rare items can still be significant
```

**Small Dataset (hundreds/thousands):**

```
min_support = 0.1 - 0.3 (10% - 30%)
Need higher frequency for statistical significance
```

**Starting Point:**

```
Start with min_support = 0.05 (5%)
Adjust based on number of rules generated:
- Too many rules (>10,000): Increase support
- Too few rules (<10): Decrease support
- Target: 100-1000 actionable rules
```

---

### **Minimum Confidence:**

**Too High (>90%):**

- Very few rules
- Miss useful patterns

**Too Low (<30%):**

- Unreliable rules
- Not actionable

**Guidelines:**

```
Typical range: 0.5 - 0.8 (50% - 80%)

Recommendation systems: 0.5 - 0.6
Strong business decisions: 0.7 - 0.8
Exploratory analysis: 0.3 - 0.5
```

---

### **Minimum Lift:**

**Lift = 1:** No association (baseline)

**Guidelines:**

```
min_lift = 1.0  (at minimum, use this)
Strong association: lift > 1.5
Very strong: lift > 2.0

Target: 1.2 - 2.0 for most applications
```

---

### **Tuning Strategy:**

```python
# Start conservative
min_support = 0.05
min_confidence = 0.6
min_lift = 1.2

# Generate rules
rules = association_rules(frequent_itemsets, 
                          metric="lift", 
                          min_threshold=min_lift)

rules = rules[(rules['confidence'] >= min_confidence) & 
              (rules['support'] >= min_support)]

# Check number of rules
n_rules = len(rules)
print(f"Number of rules: {n_rules}")

# Adjust based on results:
if n_rules > 1000:
    # Increase thresholds
    min_support *= 2
    min_confidence += 0.1
elif n_rules < 10:
    # Decrease thresholds
    min_support *= 0.5
    min_lift *= 0.8
```

## E-COMMERCE RECOMMENDATION SYSTEM

**Scenario:** Online bookstore

**Data Format:**

```python
# User-Item matrix (implicit feedback: purchased or not)
#         Book1  Book2  Book3  Book4  Book5
# User1      1      1      0      0      1
# User2      1      0      1      1      0
# User3      0      1      1      0      1
# User4      1      1      1      0      0
```

**Implementation:**

```python
from mlxtend.frequent_patterns import apriori, association_rules

# Find book associations
frequent_books = apriori(purchases, min_support=0.05, use_colnames=True)
book_rules = association_rules(frequent_books, 
                               metric="lift", 
                               min_threshold=1.5)

# Recommendation function
def recommend_books(purchased_books, rules, n=5):
    """
    Recommend books based on what user has purchased
    
    Args:
        purchased_books: set of book IDs user has purchased
        rules: association rules DataFrame
        n: number of recommendations
    
    Returns:
        List of recommended book IDs
    """
    recommendations = []
    
    for _, rule in rules.iterrows():
        antecedent = set(rule['antecedents'])
        consequent = set(rule['consequents'])
        
        # If user has antecedent books, recommend consequent
        if antecedent.issubset(purchased_books):
            recommendations.extend(list(consequent))
    
    # Remove duplicates and already purchased
    recommendations = list(set(recommendations) - purchased_books)
    
    # Sort by lift (strongest associations first)
    rec_with_lift = []
    for book in recommendations:
        lifts = rules[rules['consequents'].apply(lambda x: book in x)]['lift']
        if len(lifts) > 0:
            rec_with_lift.append((book, lifts.max()))
    
    rec_with_lift.sort(key=lambda x: x[1], reverse=True)
    
    return [book for book, lift in rec_with_lift[:n]]

# Example usage
user_purchases = {'Book1', 'Book3'}
recommendations = recommend_books(user_purchases, book_rules, n=3)
print(f"Recommended books: {recommendations}")
```

## LIMITATIONS AND CHALLENGES

### **1. Computational Complexity**

**Problem:** Exponential growth of itemsets

- n items → 2^n possible itemsets
- 50 items → 1 quadrillion itemsets

**Solutions:**

- Set appropriate min_support to prune early
- Use efficient algorithms (FP-Growth)
- Parallel processing
- Sample large datasets

---

### **2. Rare Item Problem**

**Problem:** Interesting rare items filtered out by min_support

**Example:**

```
Luxury items (caviar, champagne) have low support
But strong association: {Caviar} → {Champagne}
Filtered out by min_support threshold
```

**Solutions:**

- Use different support for different items
- Stratified mining (group by item category)
- Post-filter by lift instead of support
- Multiple support thresholds

---

### **3. Spurious Rules**

**Problem:** Statistical coincidences appear as rules

**Example:**

```
{Sales on Tuesday} → {Rain}
High confidence, but no causal relationship
```

**Solutions:**

- Use lift to filter (lift > 1)
- Domain knowledge validation
- Chi-square test for independence
- Longer observation period

---

### **4. Too Many Rules**

**Problem:** Thousands or millions of rules generated

**Solutions:**

- Increase thresholds
- Post-filtering:
    - Top-k rules by lift
    - Rules with specific items
    - Maximum rule length
    - Remove redundant rules
- Rule summarization
- Interactive visualization

---

### **5. Interpretation Difficulty**

**Problem:** Rules don't imply causation

**Example:**

```
{Diapers} → {Beer}

NOT because: Diapers cause beer purchase
ACTUALLY: Young fathers buy both
```

**Solutions:**

- Domain expert validation
- A/B testing before deployment
- Understand customer context
- Combine with other data (demographics)

---

### **6. Non-Stationary Data**

**Problem:** Purchase patterns change over time

- Seasonal variations
- Trends
- Special events

**Solutions:**

- Time-windowed analysis
- Seasonal models
- Regular re-mining
- Track rule stability over time

---

### **7. Privacy Concerns**

**Problem:** Detailed purchase patterns reveal personal information

**Solutions:**

- Aggregate data
- Differential privacy
- Anonymization
- Comply with GDPR/regulations

## EVALUATION METRICS BEYOND SUPPORT/CONFIDENCE/LIFT

### **1. Conviction**

Already covered - measures dependency strength

### **2. Zhang's Metric**

```
Zhang = (confidence(A→B) - support(B)) / max(confidence(A→B)(1-support(B)), support(B)(1-confidence(A→B)))

Range: [-1, 1]
-1: Perfect negative correlation
0: Independent
+1: Perfect positive correlation
```

### **3. Jaccard Coefficient**

```
Jaccard(A, B) = Support(A ∪ B) / (Support(A) + Support(B) - Support(A ∪ B))
              = |A ∩ B| / |A ∪ B|

Range: [0, 1]
Measures overlap between A and B
```

### **4. Certainty Factor**

```
CF(A→B) = (confidence(A→B) - support(B)) / (1 - support(B))  if conf > sup(B)
        = (confidence(A→B) - support(B)) / support(B)        if conf < sup(B)

Range: [-1, 1]
Measures increase/decrease in certainty
```

### **5. Piatetsky-Shapiro (PS)**

```
PS(A, B) = support(A ∪ B) - support(A) × support(B)

Similar to leverage
Measures deviation from independence
```

## COMMON PITFALLS

1. **Ignoring statistical significance**
    
    - Low support rules may be coincidence
    - Use chi-square test for validation
2. **Confusing correlation with causation**
    
    - Rules show association, not causation
    - Need domain knowledge to interpret
3. **Not considering item popularity**
    
    - High confidence can be misleading if consequent is very popular
    - Always check lift
4. **Setting thresholds arbitrarily**
    
    - Start with literature values
    - Adjust based on dataset characteristics
5. **Generating too many rules**
    
    - Overwhelming to interpret
    - Increase thresholds or filter post-hoc
6. **Ignoring temporal aspects**
    
    - Patterns change over time
    - Need regular updates
7. **Not validating rules**
    
    - Some high-lift rules are spurious
    - Validate with experts and A/B tests
8. **Overlooking rare but valuable patterns**
    
    - Luxury items have low support but high value
    - Consider stratified mining
9. **Poor data preparation**
    
    - Inconsistent item names
    - Missing transactions
    - Returns not handled
10. **Not considering business constraints**
    
    - Some rules not actionable
    - Implementation cost too high
    - Doesn't align with strategy

## WHEN TO USE ASSOCIATION RULES

**Use Association Rules When:**

- Analyzing transaction data (retail, e-commerce)
- Finding product bundles
- Store layout optimization
- Cross-selling strategies
- Understanding customer behavior
- Have sufficient transaction volume (100+ transactions)
- Items have reasonable frequency
- Need interpretable patterns
- Want to find unexpected associations

**Avoid Association Rules When:**

- Need personalized recommendations (use collaborative filtering)
- Have very sparse data (few transactions per user)
- Need real-time recommendations (too slow)
- Items too rare or too common
- Need sequence information (use sequential pattern mining)
- Want to predict continuous values (use regression)
- Need causal relationships (use causal inference)
